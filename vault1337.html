What Vault1337 Is

Vault1337 is a self-hosted static malware analysis platform and repository. You upload samples, store them safely, and analyse them with a large suite of format-aware static tools — all without ever executing the malware. It is built around a single guiding principle:

One analyst, one box. Vault1337 runs as a single instance for a single analyst. There is no external database, no message queue, no cloud dependency, and no team-management machinery. One account (created with createsuperuser) logs in; everything the application owns lives in one data directory you control.

What that gets you in practice:

  • Nothing to run alongside it. SQLite (in WAL mode) is the database. Upgrading is docker compose pull && up -d — no migrations to coordinate across services, nothing to back up separately.
  • Your collection is portable and durable. Point the data directory at durable storage and it survives every upgrade. Move it to another machine by copying one folder.
  • It works offline. Every external intelligence lookup is optional and degrades visibly when a key is missing or the host is airgapped — nothing hangs or fails hard.
  • Samples are stored safely. Every stored sample is obfuscated at rest so host antivirus cannot recognise and quarantine your collection, and stored read-only so the application cannot corrupt it.

The Data Directory

Everything Vault1337 writes lives under a single data directory, chosen by the VAULT_DATA_DIR environment variable. In Docker this is /data; on a native install it defaults to ~/.vault1337. This is the one thing you must put on durable storage — back it up, mount it, and your entire vault moves and upgrades with it.

PathContents
db.sqlite3The database — samples' metadata, IOCs, tags, comments, cases, analysis results
config.jsonAll application settings and API keys (written by the Management page). Mode 0600, auto-generated with a fresh SECRET_KEY on first start
samples/Stored samples, named by plaintext SHA256, obfuscated at rest and read-only
inbox/Drop-folder — files placed here are ingested automatically (see Inbox Drop Folder)
yara-rules/Your YARA .yar rule files
cyberchef/The self-hosted CyberChef build, updatable from the Management page
cache/Runtime caches (e.g. the tldextract public-suffix list)
backups/Self-contained SQLite database backups
logs/Application logs
Storage durability warning. On start, Vault1337 detects whether the data directory sits on durable storage (a bind mount or a named volume) or on a container's ephemeral overlay filesystem. If it is ephemeral — meaning your whole collection would be destroyed on the next docker compose up — it logs a loud warning and surfaces it on the Management dashboard. Always mount /data to durable storage.

Docker (Quick Start)

The fastest way to try Vault1337, with no persistent storage — the collection is lost when the container stops. Do not store anything you want to keep this way; use Docker Compose for a real deployment.

Requirements: Docker installed on your system. See the official Docker docs.
1
Pull the image
docker pull vault1337/vault1337:latest
2
Run the container
docker run -p 8000:8000 \
  -e SECRET_KEY=change-me-to-a-long-random-string \
  -e DEBUG=True \
  vault1337/vault1337:latest
3
Log in

Open http://localhost:8000 and log in with the default credentials:

Username: admin
Password: changeme123
Important: Change your password immediately via the Management page after first login.
4
Add API keys (optional)

Navigate to Management → API Keys to add keys for the intelligence integrations. See the API Keys section below.

Docker Compose (Persistent)

The recommended way to run Vault1337. A single service with everything under /data. The base compose file puts /data on a Docker named volume that survives docker compose down and image upgrades.

1
Clone the repository
git clone https://github.com/DanDreadless/Vault1337.git
cd Vault1337/Docker
2
Create the environment file

Create Docker/.env and set at minimum:

VariableDescription
SECRET_KEYLong random string — openssl rand -hex 50. If unset, one is generated into config.json on first start
ALLOWED_HOSTSComma-separated hostnames or IPs, e.g. localhost,192.168.1.10
DJANGO_SUPERUSER_PASSWORDInitial admin password (default changeme123 — change it)
VAULT_DATA_DIR=/data is set by the compose file and is the only environment variable the running application needs. API keys and app settings are not environment variables — set them from the Management page, which writes /data/config.json.
3
Start
docker compose up -d

On first start the entrypoint applies any staged restore, runs migrations, creates the superuser from the DJANGO_SUPERUSER_* env vars, and starts Gunicorn.

4
Open Vault1337

Navigate to http://localhost:8000 (or your configured HOST_PORT) and log in with admin / changeme123. Change the password immediately via the Management page.

# View logs
docker compose logs -f

# Stop the stack (your /data volume is preserved)
docker compose down

# Stop and DELETE the data volume — destroys your entire collection
docker compose down -v
5
Put /data on a specific host path (optional)

To store your vault on an external disk, NFS mount, or a known backup location instead of a named volume, set VAULT_DATA_PATH in .env and add the bind-mount override:

# .env
VAULT_DATA_PATH=/srv/vault1337

docker compose -f docker-compose.yml -f docker-compose.bindmounts.yml up -d

The host directory must be writable by UID 1001 (the container's vault user).

Docker Development

Builds the image from source and bind-mounts a host directory as /data so the database, samples, and config.json are visible from the host while you develop.

1
Start the dev stack

Run all three compose files — base, build override, and localdev overlay — from the Docker/ directory:

docker compose \
  -f docker-compose.yml \
  -f docker-compose.build.yml \
  -f docker-compose.localdev.yml \
  up --build

The build override builds the image from the repository; the localdev overlay mounts ../.docker-data (gitignored) as /data. It starts empty — the app creates its layout and generates a SECRET_KEY on first start.

2
Bring existing data in (optional)

To populate the dev data directory from an existing v1 checkout, run the migration command first, or simply drop files into the inbox and let the scanner ingest them:

python manage.py migrate_v1_to_v2 --data-dir ../.docker-data
# or drop samples into ../.docker-data/inbox/
WSL / Linux note: the container runs as vault (UID 1001). If ../.docker-data is owned by a different UID, make it writable so the container can write to it.
3
React dev server (optional, for hot-reload)

The Docker stack serves the pre-built frontend. For live frontend development with hot-reload, run the Vite dev server separately — see Local Development step 6.

Local Development

Requires Python 3.12+ and Node.js 22+. No database server to install — Vault1337 uses SQLite.

Start the React frontend (second terminal)

npm run dev

Open http://localhost:5173. The Vite dev server proxies /api/ requests to Django on port 8000.

Uploading Samples

The Home page is the ingestion hub. Five ways to bring a sample in:

MethodDescription
Upload FileDirect file upload (max 200 MB). Optionally extract a ZIP or 7z archive — with a password if needed — and each member is ingested as its own sample.
Fetch URLDownload a sample from an external URL. Private/internal IP ranges are blocked (SSRF protection).
VirusTotalDownload a sample by SHA256 hash using your VT API key.
MalwareBazaarDownload a sample by SHA256 hash using your MB API key.
Inbox drop folderDrop files into <data-dir>/inbox/ and a background scanner ingests them automatically — see Inbox Drop Folder.

All samples are stored by their plaintext SHA256 hash — original filenames are kept in the database but never used as paths on disk. Add comma-separated tags at upload time for easier organisation.

Safe at rest. Once ingested, every stored sample is written obfuscated (a fixed-key XOR, so host antivirus cannot fingerprint it) and made read-only. This is obfuscation, not encryption — its job is to stop Defender/ClamAV silently quarantining your collection, not to keep secrets. Vault1337 transparently decodes samples when a tool needs the real bytes.

Inbox Drop Folder

The data directory contains an inbox/ folder. Any file placed there is picked up by a background inbox scanner and run through the full ingest pipeline — hashed, deduplicated, enriched, obfuscated, and catalogued — then removed from the inbox. This is the fastest way to bulk-import a directory of samples: copy them in and walk away.

  • Duplicates (a sample whose SHA256 already exists) are detected and skipped, and the inbox copy is removed so it is not re-examined forever.
  • A second reconcile scanner periodically re-examines the vault and finishes analysing anything that was left under-processed (for example, if the app crashed mid-enrichment).
  • Both scanners run inside the application process — no cron, no separate worker. On a multi-worker Gunicorn deployment a lock-file election ensures only one worker scans.
  • You can also trigger a scan on demand from Management → Vault1337.

Ingest Pipeline

Every entry path — web upload, URL fetch, VirusTotal/MalwareBazaar download, archive extraction, and both scanners — runs the same ingest pipeline, so two samples with identical content always end up in an identical state regardless of how they arrived. In order:

  1. Hash — MD5, SHA-1, SHA-256, SHA-512, magic bytes, size, MIME.
  2. Deduplicate — if the SHA-256 already exists, the sample is not stored twice.
  3. SimHash — a fuzzy content fingerprint for near-duplicate clustering.
  4. Store — written to samples/<sha256> as plaintext for the enrichment step.
  5. Enrich — IOC extraction, YARA scan (auto-tagging matches), MITRE ATT&CK mapping, and a VirusTotal lookup on interactive uploads. Each step is guarded individually: a malformed sample that defeats a parser is still stored and catalogued.
  6. Obfuscate & lock — the stored file is XOR-obfuscated and set read-only.
  7. Index — added to the full-text search index.

Full-Text Search

Once a collection passes a few hundred samples, the question is rarely "what is this file called" but "which sample had that C2 domain / that mutex / that string I remember from a tool's output." The Search page answers exactly that.

  • Searches the content of the whole catalogue via a SQLite FTS5 index: sample names, tags, IOC values, saved tool output, and analyst comments.
  • Results come back most-relevant-first, each with a highlighted snippet showing the matched text in context.
  • Supports prefix search with a trailing * (e.g. cobalt*) and quoted phrases; punctuation-heavy queries (URLs, host:port) are handled safely.
  • The index stays in sync automatically — every ingest, tool run, tag change, and comment updates it. To rebuild it from scratch (e.g. after importing an old vault), run python manage.py rebuild_search_index.

Cases & Portable Archives

A case is a named grouping of samples — an investigation, a campaign, a family. It is the natural unit of analyst work: "these fourteen samples are the same intrusion." Manage cases from the Cases page.

  • Create a case, give it a description, and add samples to it — from the case page, or by selecting samples on the Vault page and choosing Add to case.
  • A sample can belong to any number of cases. Removing a sample from a case never deletes the sample itself; deleting a case never deletes its samples.

Export & Import (.vault archives)

Any case can be exported to a portable .vault archive and imported into another Vault1337 instance — the way to hand an investigation to a colleague or move it between machines.

  • The archive is an ordinary ZIP containing a manifest, per-sample metadata (tags, IOC verdicts, comments, analysis results), and the sample bytes.
  • Samples travel obfuscated, so the archive is non-executable and does not trip antivirus in transit — and importable by any instance, because the obfuscation key is fixed.
  • On import, every sample is verified against its SHA-256 before it is stored; a corrupt or tampered archive is rejected, not trusted. Nothing in the import path executes a sample.
  • Samples already present in the target vault are attached to the case without being overwritten, preserving your existing analysis.
Export a case from its detail page with Export .vault; import one with Import .vault archive on the Cases page. Import restores the analyst context as-is rather than re-analysing, so IOC true/false verdicts and comments survive the round-trip.

Sample Diffing

The Compare page (also reachable as "diff →" from clustering results) structurally compares two samples — the standard "is this a variant of that" workflow. It answers questions a raw byte diff cannot: two recompiles of the same source differ everywhere at the byte level but share their imports, sections, and strings.

For each facet it shows what is unique to sample A, unique to sample B, and how many items they share:

FacetCompared on
Imported functionsImports parsed via LIEF (PE / ELF / Mach-O)
SectionsSection names
StringsASCII and wide strings extracted from the real bytes
IOCsExtracted indicator values
ATT&CK techniquesMapped technique IDs
TagsApplied tags

A SimHash byte-similarity score is shown alongside. A PE-vs-script comparison still returns the facets it can (strings, IOCs, tags) rather than failing on the one it cannot.

SimHash Clustering

Every sample gets a SimHash fuzzy fingerprint at ingest. The Cluster view (reached via Find similar on a sample page) finds near-duplicate samples by Hamming distance — lower distance means more similar content.

  • Choose a maximum distance threshold (near-identical, similar, related, loose).
  • Results are colour-coded by similarity and sorted closest-first; the queried sample is never included.
  • Each result links straight to a structural diff against the anchor sample.

Configuration Extraction

The highest-value question about a commodity sample is often "what is its config" — the C2 addresses, the campaign/botnet ID, the mutex, the encryption key. The Config tab on a sample page runs every registered configuration extractor against the sample's decoded bytes and shows the structured result.

  • Extraction is pluggable by family: a registry of small parsers, each of which inspects a sample and returns structured config or nothing. Adding a parser for a new family is a matter of dropping one file into the extractor package.
  • Ships with two built-in heuristics: a generic network extractor (C2 host:port endpoints, URLs, and labelled fields like mutex= / botnet= from cleartext strings) and a Base64-JSON extractor (config stored as a base64-encoded JSON object, common in commodity RATs).
  • Every extractor is pure static parsing — it reads bytes, never executes them. Each result is labelled with a confidence level.
  • Results are saved to the sample's record and added to the search index.

YARA Rules & Retrohunt

Navigate to YARA in the navigation bar to manage rules with the built-in editor.

  • Create and edit rules — paste or write YARA syntax directly; rules with syntax errors fail to save with the parse error reported inline.
  • Run all rules against a single sample from its detail page using the Run YARA Rules tool. Results show the matching rule name, tags, matched string identifiers, file offsets, and match lengths.
  • Rules are stored as .yar files under <data-dir>/yara-rules/.

Retrohunt

The Retrohunt button in the rule editor runs the rule you are editing across every sample in the vault, not just one — the "I just wrote a rule, what does it already catch" workflow. Each stored sample is decoded before scanning, and the results list every match with its sample and rule name. Because rules and samples are both local, the hunt is fast and needs no external service.

No rules are included by default — Vault1337 ships with a starter set seeded into <data-dir>/yara-rules/ on first use, but building a useful rule set is up to you. Add your own .yar files via the web editor or by placing them in that directory.

IOC Management

Navigate to IOCs to view all extracted indicators of compromise across all samples.

  • Run the Extract IOCs tool on a sample to populate its IOCs.
  • Each IOC is linked back to the sample it came from.
  • Mark IOCs as true positive or false positive using the toggle on the IOC page.
  • Filter the list by classification (true / false / unreviewed) and by IOC type using the filter chips above the table.
  • Search by IOC value using the search box.

Auto-Enrichment

When the Extract IOCs tool runs, newly extracted ip and domain IOCs are automatically enriched against threat intel APIs in a background thread — the tool response returns immediately while enrichment continues asynchronously.

  • IP IOCs — queried against VirusTotal and AbuseIPDB concurrently.
  • Domain IOCs — queried against VirusTotal.
  • An IOC is marked true positive if the VT malicious count meets the threshold or the AbuseIPDB score meets the threshold. It is marked false positive if all sources return clean results.
  • RFC1918 private IPs (10/8, 172.16/12, 192.168/16) are saved as IOCs (useful for lateral movement analysis) but auto-marked false positive — they are never sent to external APIs.
  • Loopback, link-local, multicast, and other reserved IP ranges are discarded entirely at extraction time.

The enrichment column in the IOC table shows results as VT: x/y (malicious/total engines) and AIPDB: z% (abuse score). A dash (—) is shown for IOC types not yet enriched or where no API key is configured.

Bulk Actions

Each row in the IOC table has a checkbox. Selecting one or more rows reveals a bulk action bar with two options:

  • Export STIX — exports the selected IOCs as a STIX 2.1 bundle (see STIX 2.1 Export below).
  • Delete — permanently removes the selected IOCs. Requires a confirmation click to prevent accidental deletion (staff only).

Manual Override & Re-Enrichment

  • Manually toggling an IOC's true/false status sets an overridden flag, shown as a grey pill next to the status badge. Subsequent background enrichment runs will not overwrite a manually overridden IOC.
  • A Re-enrich button is available on each ip and domain row. Clicking it triggers a synchronous re-enrichment (both VT and AbuseIPDB in parallel) and immediately refreshes the row. This clears the overridden flag so fresh data is applied.

IP & Domain Intelligence

IP Check

Navigate to IP Check to query an IP address against multiple threat intelligence sources simultaneously.

  • Displays a colour-coded verdict banner (Malicious / Suspicious / Clean / Unknown) derived from all sources combined.
  • Individual source cards: VirusTotal (detection count + link), AbuseIPDB (score bar + usage type + ISP), Spur (tunnel/tag badges), Shodan (open ports, hostnames, CVEs).
  • Source cards with missing API keys are sorted to the bottom of the page.
  • Private, loopback, and reserved IP ranges are rejected before any external query is made.

Domain Check

Navigate to Domain Check to query a domain against VirusTotal and WHOIS.

  • Displays a verdict banner derived from VirusTotal detections.
  • Individual source cards: VirusTotal (detection count, category badges + link), WHOIS (registrar, creation/expiry dates, name servers, status badges), Passive DNS (unique IPs seen resolving to this domain).
  • Cards with missing API keys are sorted to the bottom.

Tags

  • Add comma-separated tags when uploading a sample, or add/remove tags from any sample on its detail page.
  • Ingest applies tags automatically — the file extension, the source it came from, and one yara:<rule> tag per matching YARA rule.
  • All tags are stored in lowercase.
  • Filter the Vault page by filename, hash, or tag with its search box and the file-type chips; for content search across tool output and IOCs, use Full-Text Search.

Analyst Notes

Analyst notes let you attach structured commentary to any sample — useful for documenting findings, sharing context with team members, and capturing initial triage decisions.

  • Open a sample and select the Notes tab on the sample detail page.
  • Each note has a title, a body (supports Markdown formatting), and a type label.
  • Note types: note (general observation), hypothesis (a working theory), ioc_context (indicator context), and verdict (final assessment).
  • Notes are preserved across sessions and included in the full-text search index.
  • Notes appear in the PDF report export under the Analyst Notes section.
Note bodies are rendered as Markdown in the UI — use headings, bullet lists, code blocks, and bold/italic formatting to structure your findings.

Sample Report

Every sample has a structured JSON report available at GET /api/v1/files/{id}/report/. The report aggregates all stored analysis data in a single response — no new analysis is run.

The report includes:

  • File metadata — name, size, magic bytes, MIME type, upload date, uploader
  • Hashes — MD5, SHA1, SHA256, SHA512
  • Tags — all tags applied to the sample
  • VirusTotal summary — detection counts (malicious / suspicious / undetected / harmless), threat label, threat category, scan date, and known names
  • IOCs — all confirmed (true positive) IOCs grouped by type
VirusTotal data is only included if a VT enrichment has previously been run on the sample (automatically on upload if a VT API key is configured, or manually via the VT Enrich action).

PDF Report Export

Every sample detail page has a PDF Report button on the Info tab. Clicking it generates a professional A4 PDF report entirely in-browser — no server round-trip is required beyond the initial page load.

The PDF is built from data already loaded on the page and downloads instantly with the filename:

MITRE ATT&CK Mapping

Each sample has a dedicated ATT&CK tab on the sample detail page showing matched MITRE ATT&CK techniques. A summary card is also shown on the Info tab.

  • Click Map Techniques to scan the sample's stored analysis results and linked IOC types for known MITRE ATT&CK technique indicators.
  • Results are displayed as colour-coded tactic badges arranged by tactic. Each badge shows the technique ID and name; expanding a badge shows the matched indicator evidence.
  • Click a badge to open the official MITRE ATT&CK technique page in a new tab.
  • Mapping covers 27 techniques across 10 tactics including execution, persistence, defence evasion, discovery, lateral movement, and impact.
  • Detection uses two sources: (1) regex pattern matching against saved AnalysisResult text (excluding IOC extractor output to prevent section-header false positives), and (2) direct mapping from linked IOC types (e.g. scheduled_task → T1053, win_persistence → T1547).
  • Results are persisted on the sample record and can be re-mapped at any time via the Re-map button.

STIX 2.1 Export

Vault1337 supports exporting threat intelligence in STIX 2.1 format via the stix2 library.

Sample STIX Export

On the sample detail page, the Info tab has an Export STIX button. Clicking it downloads a STIX 2.1 bundle for that sample containing:

  • A Vault1337 Identity SDO as the bundle author.
  • A File SCO with MD5, SHA-1, and SHA-256 hashes.
  • An Indicator SDO with a hash-based STIX pattern.
  • STIX objects for every linked IOC — IPv4Address, DomainName, URL, EmailAddress, WindowsRegistryKey SCOs with associated Indicator SDOs; Vulnerability SDOs for CVE IOCs; Note SDOs for persistence and other types.

Bulk IOC STIX Export

On the IOC page, select one or more IOCs using the row checkboxes, then click Export STIX in the bulk action bar. The download contains STIX objects for all selected IOCs, without being tied to a specific sample.

CyberChef

Navigate to CyberChef in the navigation bar for a full, self-hosted copy of CyberChef — GCHQ's "Cyber Swiss Army Knife" — embedded directly in Vault1337 for decoding, deobfuscating, and transforming data during analysis (Base64, hex, XOR, gzip, magic detection, and hundreds of other operations).

  • Served entirely from your own instance out of the data directory (<data-dir>/cyberchef/) — no internet access is required at runtime and nothing you paste leaves your box.
  • Check the bundled version and update it to the latest GitHub release with one click from Management → CyberChef.

Integrity & Backups

Integrity Verification

Because samples are stored by their plaintext SHA-256, the store is self-describing: every file can be checked against its own name. Integrity verification re-hashes every stored sample and reports drift — corruption, truncation, antivirus tampering, and missing or orphaned files.

  • Run it from Management → Vault1337, or on the command line with python manage.py verify_integrity.
  • Each sample is reported as OK, corrupt (matches neither its plaintext nor obfuscated hash), or missing.

Database Backups

Create, download, and restore self-contained SQLite database backups from the Management page.

  • A backup is a single self-contained SQLite file written to <data-dir>/backups/ — download it and keep it anywhere.
  • Restores are staged: because the database cannot be swapped under a live connection, a requested restore is applied cleanly on the next start-up, before anything opens the database.
  • Samples are the source of truth — restoring an older database and then running a reconcile scan re-catalogues any sample files still present.
The whole vault is one directory. For a full off-box backup, snapshot or copy <data-dir> — database, samples, config, rules and all.
